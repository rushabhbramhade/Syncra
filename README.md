## Overview

**The problem.**
Building an AI agent that can actually *reach* people — not just sit in a web chat widget — usually means stitching together separate SMS gateways, WhatsApp Business APIs, RCS providers, delivery-status webhooks, and template approval workflows. Each channel has its own auth model, rate limits, and formatting rules, which turns a simple "send a message" feature into weeks of integration work. On top of that, scaffolding the surrounding SaaS shell (auth, dashboard, database, deployment) from scratch is slow and repetitive.

**Who it's for.**
Syncra is built for indie developers, AI tinkerers, and early-stage SaaS founders who want to ship a working, message-capable AI agent product in hours instead of weeks — without becoming telecom integration experts first.

**Why it was built.**
Syncra exists to demonstrate a faster path from idea to production: use an agentic CLI (**Antigravity 2.0**) to generate the full-stack scaffolding (frontend, API routes, database schema) directly from natural-language prompts, then bolt on **Sent.dm's** unified messaging API so the agent can speak SMS, WhatsApp, and RCS through one integration instead of three. The result is a personal AI agent that can be reached and replied to wherever the user already is.

## Features

* AI agent that understands and responds to user requests (assumed)
* Unified outbound messaging across SMS, WhatsApp, and RCS via Sent.dm
* Automatic channel routing with fallback when a channel is unavailable
* Delivery-status tracking via Sent.dm webhooks (assumed)
* Full app scaffold (frontend + API + schema) generated by Antigravity 2.0 CLI
* Next.js + React dashboard for managing agent conversations (assumed)
* Template-based messaging for OTPs, confirmations, and replies
* Environment-based configuration for API keys and database credentials

## Architecture

```text
Client (Next.js / React UI)
  ↓
Next.js API Routes (Service Layer)
  ↓
AI Agent Orchestration
  ↓
Sent.dm Messaging API   +   PostgreSQL Database
  ↓
Delivery Webhooks → Status Updates (assumed)
```

The client talks only to the Next.js service layer — it never calls Sent.dm directly. The service layer runs agent logic, decides what to send, and hands the message off to Sent.dm, which auto-routes it to the best available channel per recipient. Delivery confirmations flow back in through webhooks (assumed) so the dashboard can reflect real message status.

## Getting Started

### Prerequisites
- Node.js 18+ and npm/pnpm
- A Sent.dm account (verified email + phone, ~2 min signup)
- PostgreSQL instance (local or hosted) (assumed)

### Installation
```bash
git clone https://github.com/your-org/syncra.git
cd syncra
npm install
```

### Environment Variables
```env
SENT_API_KEY=your_sent_dm_api_key
DATABASE_URL=postgresql://user:password@localhost:5432/syncra   # assumed
NEXT_PUBLIC_APP_URL=http://localhost:3000
```

### Run locally
```bash
npm run dev
```

## Usage

```ts
import SentDm from '@sentdm/sentdm';

const client = new SentDm(); // reads SENT_API_KEY from env

await client.messages.send({
  to: ['+14155551234'],
  template: { id: 'agent_reply_template' },
  channel: ['sms', 'whatsapp', 'rcs'],
});
```
